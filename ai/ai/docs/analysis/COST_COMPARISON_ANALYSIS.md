```
전제:
- 활성 사용자: 2,000명
- 일일 사용률: 30% (600명/일)
- 1인당 평균 조회: 4회/일

계산:
- 일일 요청: 600명 × 4회 = 2,400회/일
- 월간 요청: 2,400회 × 30일 = 72,000회/월
```

### 12.4 피크 시간대 분석

**SSAFY 특성상 예상 피크**:

| 시간대 | 예상 트래픽 | 비고 |
|--------|-----------|------|
| 09:00~10:00 | 높음 (20%) | 아침 출석, 오늘의 운세 |
| 12:00~13:00 | 중간 (15%) | 점심시간 |
| 18:00~19:00 | 높음 (25%) | 퇴실 후 확인 |
| 21:00~23:00 | 중간 (15%) | 저녁 여가시간 |
| 기타 | 낮음 (25%) | 분산 |

**순간 피크 계산 (시나리오 B 기준)**:

```
일일 600회 중 피크 시간대 25% 집중
→ 피크 1시간: 150회
→ 분당: 2.5회
→ 초당: 0.04회 (40초에 1회)

vLLM 처리 능력 (8B FP16):
- 평균 응답 시간: 3~5초
- 동시 처리: 1~2 요청 (배치)
→ 분당 최대 12~20회 처리 가능

결론: 피크 시에도 충분한 여유 ✅
```

### 12.5 이벤트성 피크 (최악의 시나리오)

**"새해 첫날 운세" 같은 이벤트 시**:

```
전제:
- 전체 사용자의 50%가 1시간 내 접속
- 2,000명 × 50% = 1,000명
- 1인당 2회 조회

계산:
- 1시간 총 요청: 2,000회
- 분당: 33회
- 초당: 0.55회

vLLM 처리:
- 분당 12~20회 처리 가능
- 초과분: 큐잉 발생 (13~21회/분 대기)
- 예상 대기 시간: 30초~2분

대응 방안:
1. Redis 캐싱 활용 (동일 생년월일 캐시)
2. 요청 제한 (Rate Limiting)
3. 스케일업 (g5.2xlarge로 임시 전환)
```
