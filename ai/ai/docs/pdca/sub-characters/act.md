## 1. 성공 패턴 정리

### 1.1 Few-shot 프롬프팅 전략

#### 효과적인 Few-shot 구성 요소

| 요소 | 설명 | 효과 |
|------|------|------|
| 명시적 규칙 | "절대 금지", "반드시 사용" 등 강한 어조 | 규칙 준수율 30% 향상 |
| 자동 변환 예시 | "합니다 → 해요" 패턴 10개 이상 | 합니다체 사용 50% 감소 |
| 올바른 예시 30개 | 실제 응답에 가까운 다양한 문장 | 패턴 학습 효과 극대화 |
| 틀린 예시 10개 | ❌/✅ 비교 형식 | 명확한 경계 설정 |
| 최종 점검 섹션 | 응답 생성 전 자가 검토 지시 | 마지막 단계 필터링 |

#### 프롬프트 구조 템플릿

```
/no_think
[캐릭터 인사/소개 - 1-2문장]

[핵심 규칙 - 엄격히 준수]
✅ 필수 어미: ...
❌ 금지 어미: ...

<말투 규칙 - 반드시 암기>
❌ 절대 금지: [구체적 패턴 나열]
✅ 오직 사용: [구체적 패턴 나열]

<자동 변환 - X→Y>
- "패턴A" → "패턴B" (예시 포함)
[10개 이상]
</자동 변환>

<올바른 문장 예시 30개>
1. "..."
2. "..."
...
</올바른 문장 예시>

<틀린 문장 예시>
❌ "..." → ✅ "..."
[10개 이상]
</틀린 문장 예시>

<character>
[캐릭터 성격 2-3문장]
</character>

<최종 점검 - 응답 생성 전 반드시 확인>
1. [체크포인트 1]
2. [체크포인트 2]
...

⚠️ 경고: [규칙 위반 시 경고 메시지]
</최종 점검>
```

### 1.2 합니다체 → 해요체 변환 노하우

#### 변환 규칙 정리

| 합니다체 | 해요체 | 예시 |
|----------|--------|------|
| ~합니다 | ~해요 | 환영합니다 → 환영해요 |
| ~입니다 | ~이에요/예요 | 공주입니다 → 공주예요 |
| ~있습니다 | ~있어요 | 기다리고 있습니다 → 기다리고 있어요 |
| ~없습니다 | ~없어요 | 문제없습니다 → 문제없어요 |
| ~됩니다 | ~돼요 | 가능합니다 → 가능해요 (or 돼요) |
| ~겠습니다 | ~드릴게요/할게요 | 도와드리겠습니다 → 도와드릴게요 |
| ~바랍니다 | ~바라요 | 잘 되기 바랍니다 → 잘 되기 바라요 |
| ~좋습니다 | ~좋아요 | 아주 좋습니다 → 아주 좋아요 |

#### LLM이 합니다체를 선호하는 맥락

1. **공식적 맥락**: "운세", "예측", "분석" 관련 문장
2. **정보 전달**: 객관적 사실 전달 시
3. **격식체 유도**: "왕국", "공주" 등 격식 키워드 존재 시

#### 해결 전략

1. **프롬프트 강화**: 맥락별 변환 예시 추가
2. **최종 점검 지시**: "합니다체가 하나라도 있으면 수정 후 응답"
3. **Postprocessor 백업**: 프롬프트로 해결 안 될 경우 후처리 변환

## 2. 캐릭터별 학습 내용

### 2.1 청운 (CHEONGWOON)

**상태**: ✅ 완료 (90점)

**성공 요인**:
- 하오체 어미가 명확하여 LLM이 쉽게 학습
- 시적 비유 키워드 ("물", "바람", "달")가 톤 유지에 효과적

**주의사항**:
- "허허", "호호" 등 감탄사가 캐릭터성 유지에 중요
- 너무 긴 응답 시 합니다체 혼입 가능성 있음

### 2.2 화린 (HWARIN)

**상태**: ✅ 완료 (90점)

**성공 요인**:
- 비즈니스 용어 ("공짜", "할인", "단골")가 캐릭터 톤 유지
- 해요체가 자연스러운 캐릭터라 LLM 적응 용이

**주의사항**:
- "어머", "세상에" 등 감탄사 패턴 유지 필요
- 나른한 톤 유지를 위해 문장 길이 조절 권장

### 2.3 카일 (KYLE)

**상태**: ✅ 완료 (90점)

**성공 요인**:
- 도박 비유 ("패", "올인", "조커")가 캐릭터성 강화
- 반말+존댓말 혼용이 자연스럽게 구현됨

**주의사항**:
- 테스트 코드의 반말 regex 확장 필요 (P3)
- "~거든", "~잖아" 패턴 추가 검증 필요

### 2.4 엘라리아 (ELARIA)

**상태**: ⚠️ 조건부 완료 (80점, 허용 범위)

**도전 과제**:
- 합니다체 완전 제거가 어려움 (LLM 격식체 선호)
- "운세" 맥락에서 특히 합니다체 빈도 증가

**해결책**:
1. Few-shot 30개 예시 + 틀린 예시 10개
2. "최종 점검" 섹션으로 자가 검토 유도
3. ⚠️ 경고 메시지로 규칙 위반 강조

**현재 수치**:
- 해요체 비율: 평균 86% (목표 80% 이상)
- 합니다체 빈도: 평균 1개 (목표 2개 이하)

    """프로덕션 배포 전 품질 검증"""

### 4.2 P3: 엘라리아 Postprocessor 변환 (선택)

**조건**: 해요체 비율 80% 미만 시 구현

**현재**: 86% 달성으로 미구현 결정

**구현 시**:
```python
def convert_habnida_to_haeyo(text: str) -> str:
    """합니다체 → 해요체 자동 변환"""
    replacements = [
        (r"합니다", "해요"),
        (r"입니다", "이에요"),
        # ...
    ]
    for pattern, replacement in replacements:
        text = re.sub(pattern, replacement, text)
    return text
```

### 4.3 P3: 카일 반말 regex 확장

**현재 패턴**: `(해\.|야\.|지\.|거든\.)`

**확장 제안**: `(해\.|야\.|지\.|거든\.|잖아|이야|거야|네\.|어\.)`

### 핵심 학습

1. **Few-shot은 30개 이상이 효과적**: 10개 미만은 패턴 학습 부족
2. **틀린 예시가 효과적**: ❌→✅ 비교 형식이 경계 설정에 유용
3. **최종 점검 섹션 필수**: 응답 생성 전 자가 검토 지시가 품질 향상에 기여
4. **합니다체 완전 제거는 어려움**: 80% 이상 해요체 비율을 현실적 목표로 설정
5. **Postprocessor는 백업 수단**: 프롬프트 최적화 우선, 안 되면 후처리
